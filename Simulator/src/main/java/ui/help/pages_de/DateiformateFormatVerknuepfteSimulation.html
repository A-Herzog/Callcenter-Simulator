Das Basis-Element für verkettete Simulationsmodelle heißt <VerknuepfteSimulation> und muss ein Attribut Basisverzeichnis besitzen, welches angibt auf welches Verzeichnis sich relative Angaben beziehen sollen. Des Weiteren kann es ein Attribut InitialerUebertrag besitzen, welches den Namen einer Statistikdatei angibt, aus der der Übertrag zum ersten Tag hin entnommen werden soll.

Jedes verkettete Simulationsmodell muss ein odere mehrere Elemente vom Typ <VerknuepftTag> besitzen. Jedes dieser Elemente stellt einen bestimmten Simulationstag dar. Außerdem kann optional ein Element vom Typ <VerknuepftInitialerBestand> existieren.

Informationen zu den einzelnen Simulationstagen

Jedes <VerknuepftTag>-Element muss folgende drei Unterelemente in dieser Reihenfolge besitzen:

  • <VerknuepftModell>: Gibt das zu simulierende Modell an (Dateiname der xml-Modelldatei). Ist der Eintrag leer, so wird das Modell vom Vortag verwendet. Für den ersten Tag darf der Eintrag nicht leer sein. Enthält der Eintrag einen Verzeichnistrenner, so wird der Pfad als absoluter Pfad interpretiert, ansonsten wird angenommen, dass es sich um eine Datei im angegebenen Basisverzeichnis handelt.
  • <VerknuepftStatistik>: Gibt den Dateinamen für die Statistikausgabe an. Ist der Eintrag leer, so werden die Ergebnisse des Simulationstages nicht gespeichert und nur zur Verknüpfung mit dem nächsten Tag verwendet. Existiert die hier angegeben Statistikdatei bereits zu Beginn der Simulation, so wird sie überschrieben. Enthält der Eintrag einen Verzeichnistrenner, so wird der Pfad als absoluter Pfad interpretiert, ansonsten wird angenommen, dass es sich um eine Datei im angegebenen Basisverzeichnis handelt.
  • <VerknuepftUebertraege>: Dieses Element kann einen oder mehrere Kindelemente vom Typ <VerknuepftUebertrag> besitzen. Jedes <VerknuepftUebertrag>-Element muss die beiden Attribute Typ und Wert besitzen. Die Übertragselemente geben an, wieviele der endgültigen Warteabbrecher vom Vortag im aktuellen Tag zu den Erstanrufern hinzugenommen werden sollen. Ist das <VerknuepftUebertraege>-Element leer, so erfolgt kein Übertrag. Besitzt das <VerknuepftUebertraege>-Element genau ein <VerknuepftUebertrag>-Kindelement, so muss bei diesem das Typ-Attribut leer sein. Das Wert-Attribut gibt dann den Anteil der zu übertragenden Warteabbrecher für alle Kundentypen pauschal an. Besitzt das <VerknuepftUebertraege>-Element mehrere <VerknuepftUebertrag>-Kindelemente, so erfolgt die Übertragung der Warteabbrecher kundentypspezifisch. Jedes <VerknuepftUebertrag>-Element gibt dabei den Wert für einen Kundentyp an. Der jeweilige Kundentyp wird im Typ-Attribut festgelegt. Wert gibt den Anteil der zu übertragenden Warteabbrecher an. Es müssen nicht für alle Kundentypen <VerknuepftUebertrag>-Elemente geben bzw. es ist nicht notwendig, Elemente mit Wert="0" anzulegen, wenn für einen bestimmten Typ kein Übertrag erfolgen soll. Alle per Typ definierten Kundentypen müssen jedoch in dem verwendeten Simulationsmodell existieren. Das <VerknuepftUebertrag>-Element kann optional noch Unterelemente vom Typ <VerknuepftUebertragNeuerKundentyp> enthalten. Jedes <VerknuepftUebertragNeuerKundentyp>-Element muss die beiden Attribute Typ und Wert besitzen, die angeben, mit welcher Rate sich der Typ eines Kunden zu diesem anderen Typ ändert. Besitzt ein <VerknuepftUebertrag>-Element keine <VerknuepftUebertragNeuerKundentyp>-Unterelemente, so findet keine Kundentypänderung statt.

Informationen zum manuellen Übertrag in Tag 1 hinein

Das optionale <VerknuepftInitialerBestand>-Element kann beliebig viele <VerknuepftInitialerBestandKundentyp>-Elemente beinhalten. Jedes <VerknuepftInitialerBestandKundentyp>-Element muss ein Name-Attribut besitzen, das angibt auf welchen Kundentyp sich das Element bezieht und muss als Inhalt eine nichtnegative ganze Zahl enthalten, die angibt, wie viele zusätzliche Erstanrufer als manueller Übertrag aus dem vorherigen Tag in das Modell für den ersten verketteten Tag hinzugefügt werden sollen.
