Das Basis-Element für Callcenter-Modell-Dateien heißt <CallcenterModell>.

Allgemeine Angaben

Das <CallcenterModell> Element enthält folgende Unterelemente, die allgemeine Angaben zu dem Callcenter-Modell enthalten. Jedes der folgenden Unterelemente muss genau einmal vorhanden sein (sofern nicht anders angegeben):

  • <ModellSimulatorVersion>: Gibt die Version des Simulators, mit dem das Modell erstellt wurde. Ist hier ein größerer Wert eingetragen als die Versionummer der verwendeten Version, so warnt der Simulator beim Laden des Modell über die grafische Benutzeroberfläche.
  • <ModellName>: Enthält den Namen des Modell. Der Name dient nur Identifikation des Modells durch den Nutzer.
  • <ModellDatum>: Dieses Element ist optional. Es gibt das Datum des Tages an, den das Simulationsmodell repräsentieren soll. Das Format für die Datumsangaben ist MM/DD/YYYY, aber beim Laden werden auch lokale Datumsformate erkannt.
  • <ModellBeschreibung>: Enthält eine Beschreibung des Modells. In der Beschreibung können beliebige Informationen für den Nutzer angegeben werden.
  • <ModellMaxWarteschlangenLaenge>: Gibt die Maximallänge der Warteschlange an. Befinden sich bereits so viele Kunden in der Warteschlange, wie hier angegeben, so werden weitere eintreffende Kunden abgewiesen. Kunden, die sich bereits in Bedienung befinden, werden dabei nicht mitgezählt.
  • <ModellSimulierteTage>: Gibt die Anzahl an zu simulierenden Tagen an. Je mehr Tage simuliert werden, desto statistisch stabiler sind die Ergebnisse.
  • <ModellBevorzugteSchichtlaenge>: Werden Agentengruppen in Form von Agentenzahlen pro Halbstundenintervall und nicht in Form von einer fixen Zahlen an Agenten mit konkreten Arbeitszeiten angegeben, so müssen vor Beginn der Simulation konkrete Agenten mit konkreten Arbeitszeiten generiert werden. Um die Eigenschaften des echten Callcenter-Systems möglichst realistisch nachzubilden, werden keine Agenten mit Halbstunden-Arbeitszeiten generiert, da dies aufgrund von Überschneidungen zu einer Überschätzung der real verfügbaren Arbeitszeiten führen würde. Stattdessen wird versucht, Agenten mit der hier angegebenen Schichtlänge zu generieren. Dieser globale Vorgabewert kann über ein Datenfeld mit identischem Namen pro Agentengruppe überschrieben werden.
  • <ModellMinimaleSchichtlaenge>: Über dieses optionale Element kann zusätzlich zu dem <ModellBevorzugteSchichtlaenge>-Element eine minimale Schichtlänger der Arbeitszeiten der Agenten festgelegt werden. Fehlt dieses Element, so wird eine minimale Schichtlänge von einem Intervall (was also keiner Einschränkung nach unten entsprecht) angenommen. Dieser globale Vorgabewert kann über ein Datenfeld mit identischem Namen pro Agentengruppe überschrieben werden.
  • <ModellServiceLevel>: Während der Simulation wird erfasst, für wie viele Anrufer das gewünschte Service-Level erreicht wurde. Das Service-Level ist der Anteil an Anrufern, der höchstens eine gewisse Anzahl an Sekunden gewartet hat. Über dieses Feld kann angegeben werden, welche Sekundenzahl als Grenze für die Erreichung des Service-Level angesetzt werden soll.
  • <ModellProduktivitaet>: In diesem Feld werden 48 durch ";" getrennte Werte erwartet, die die Produktivität der Agenten im Tagesverlauf angeben. Die verfügbare Arbeitsleitung pro Halbstundenintervall bei Agentengruppen, die über eine Arbeitsleitungsverteilungskurve oder über die Nachmodellierung der Kundenankünfte definiert sind, wird mit diesem Produktivitätswert multipliziert. Diese globale Produktivität kann optional pro Callcenter und pro Agentengruppe überschrieben werden.
  • <ModellPlanungsaufschlag>: In diesem Feld werden 48 durch ";" getrennte Werte erwartet, die die angeben, um wie viel höher die planerischer Agentenanzahl sein soll als im Modell im Tagesverlauf angeben. Die verfügbare Arbeitsleitung pro Halbstundenintervall bei Agentengruppen, die über eine Arbeitsleitungsverteilungskurve oder über die Nachmodellierung der Kundenankünfte definiert sind, wird mit diesem Produktivitätswert multipliziert und in der Statistik gespeichert. Die Agentenanzahl inkl. diesem Aufschlag wird nicht für die Simulation selber verwendet. Dieser globale Aufschlag kann optional pro Callcenter und pro Agentengruppe überschrieben werden.

Kundentypen

Das <CallcenterModell> Element kann beliebig viele Elemente (mindestens jedoch eins) vom Typ <ModellKundentyp> enthalten.

Attribute des ModellKundentyp-Elements

  • Name: Gibt den Namen des Kundentyps an. Dieser Name wird z.B. bei der Definition von Skill-Levels verwendet. Das Name-Attribut muss existieren.
  • Aktiv: Gibt an, ob die im Folgenden definierte Kundengruppe in der Simulation verwendet werden soll. Aktiv kann die Werte 0 und 1 annehmen. Das Attribut ist optional; wenn es fehlt, wird "1" angenommen.
  • BelegtLeitung: Gibt an, ob die Kunden der im Folgenden definierten Kundengruppe während des Wartens und während der Bearbeitung eine Telefonleitung belegen (wie dies bei Anrufen der Fall ist) oder nicht (wie bei E-Mails usw.). BelegtLeitung kann die Werte 0 und 1 annehmen. Das Attribut ist optional; wenn es fehlt, wird "1" angenommen.
  • ServiceLevel: Ist dieses optionale Attribut vorhanden, so wird in der Statistik für diesen Kundentyp nicht der globale Service-Level Sekundenwert verwendet, sondern der hier angegebene Wert.

Unterelemente des ModellKundentyp-Elements

Das <ModellKundentyp>-Element enthält folgende Unterelemente. Jedes der folgenden Unterelemente muss genau einmal vorhanden sein:

  • <KundenErstanrufer>: Dieses Element enthält genau die zwei Unterelemente ErstanruferAnzahl und ErstanruferVerteilung.
    • <ErstanruferAnzahl> gibt die Anzahl an Erstanrufern des Kunden dieses Typs pro Tag an. Als optionalen Parameter kann das Element ein Standardabweichung-Attribut besitzen. Dieses gibt, wenn vorhanden, an, mit welche Standardabweichung (normalverteilt betrachtet) die Anzahl an Erstanrufern pro Tag schwankt. Fehlt das Attribut, so wird hier 0 angenommen.
    • <ErstanruferVerteilung> enthält 24, 48 oder 96 durch Semikolons getrennte Werte, die den Anteil der Erstanrufer dieses Kundentyps pro Stunden-, Halbstunden- oder 15-Minutenintervall angibt. Die konkrete Größe der Werte ist dabei nicht von Bedeutung, sondern nur das relative Verhältnis der Größen der Werte untereinander. Die Gesamtzahl an Erstanrufern des Kundentyps pro Tag an wird ausschließlich über das Element ErstanruferAnzahl bestimmt.
  • <KundenScore>: Dieses Element hat keinen Inhalt, muss aber die folgenden drei Attribute besitzen: Basis, ProWartesekunde und Weiterleitung. Die drei Attribute geben den Basis-Scorewert des Kundentyps, den Wert, um den die Score pro Wartesekunde erhöht wird, und den Score-Aufschlag, den ein Kunde, der weitergeleitet wird, erhält, an.
  • <KundenWartezeittoleranzVerteilung>: Es muss entweder dieses Element oder das Element <KundenWartezeittoleranzVerteilungLang> oder das Element <KundenWartezeittoleranzSchaetzung> vorhanden sein. Mehrere gleichzeitig sind nicht zulässig.
    In diesem Element wird eine Verteilung angegeben, gemäß derer die Wartezeittoleranzen der Kunden bestimmt werden. Werden empirische Daten angegeben, so werden diese auf eine Gesamtdauer von einer Stunde skaliert.
  • <KundenWartezeittoleranzVerteilungLang>: Es muss entweder dieses Element oder das Element <KundenWartezeittoleranzVerteilung> oder das Element <KundenWartezeittoleranzSchaetzung> vorhanden sein. Mehrere gleichzeitig sind nicht zulässig.
    In diesem Element wird eine Verteilung angegeben, gemäß derer die Wartezeittoleranzen der Kunden bestimmt werden. Werden empirische Daten angegeben, so werden diese auf eine Gesamtdauer von 10 Tagen skaliert.
  • <KundenWartezeittoleranzSchaetzung>: Es muss entweder dieses Element oder das Element <KundenWartezeittoleranzVerteilung> oder das Element <KundenWartezeittoleranzVerteilungLang> vorhanden sein. Mehrere gleichzeitig sind nicht zulässig.
    Dieses Element hat keinen Inhalt, muss aber die folgenden drei Attribute besitzen: Wartezeit, Abbruchwahrscheinlichkeit und Korrekturwert. Das Wartezeit-Attribut nimmt eine Zeitangabe in der Form hh:mm:ss.nnn auf und gibt die mittlere Wartezeit der Kunden an. Das Abbruchwahrscheinlichkeit-Attribut gibt die gemessene Abbruchrate an, aus der die Wartezeittoleranz hochgerechnet werden soll. Das Korrekturwert-Attribut gibt an, um wie viel die hochgerechnete Wartezeittoleranz nach oben oder unten korrigiert werden soll und nimmt eine Zeitangabe in der Form hh:mm:ss.nnn oder -hh:mm:ss.nnn auf.
  • <KundenWiederholungen>: Dieses Element muss genau die folgenden neun Unterelemente enthalten:
    • <WiederholungenAbstaendeVerteilung>: Gibt die Verteilung der Abstände zwischen dem Auflegen eines Kunden und einem weiteren Anrufversuch an.
    • <WiederholungenWahrscheinlichkeitNachBlockierungErsterVersuch>: Gibt die Wahrscheinlichkeit an, mit der Kunden nach der ersten Blockierung (=Warteschlange zum Anrufzeitpunkt voll) einen weiteren Anruf tätigen.
    • <WiederholungenWahrscheinlichkeitNachBlockierungWeitereVersuche>: Gibt die Wahrscheinlichkeit an, mit der Kunden, die bereits zu vor einmal blockiert wurden oder aufgegeben haben, nach einer Blockierung (=Warteschlange zum Anrufzeitpunkt voll) einen weiteren Anruf tätigen.
    • <WiederholungenWahrscheinlichkeitNachAbbruchErsterVersuch>: Gibt die Wahrscheinlichkeit an, mit der Kunden, die im ersten Anrufversuch das Warten aufgegeben haben, später einen weiteren Anruf tätigen.
    • <WiederholungenWahrscheinlichkeitNachAbbruchWeitereVersuche>: Gibt die Wahrscheinlichkeit an, mit der Kunden, die bereits mindestens zum zweiten Mal anrufen und das Warten aufgegeben haben, später einen weiteren Anruf tätigen.
    • Die vier Elemente <WiederholungenKundentypNachBlockierungErsterVersuch>, <WiederholungenKundentypNachBlockierungWeitereVersuche>, <WiederholungenKundentypNachAbbruchErsterVersuch> und <WiederholungenKundentypNachAbbruchWeitereVersuche> müssen jeweils für jeden Kundentyp ein <WiederholungenNeuerKundentyp> besitzen. <WiederholungenNeuerKundentyp> besitzt ein Attribut Name, welches angibt, auf welchen Kundentyp es sich bezieht und gibt im Element-Inhalt an, mit welche Rate ein Kunde seinen Typ nach einem Abbruch in den angegebenen Typ ändert. Haben alle <WiederholungenNeuerKundentyp>-Elemente den Inhalt "0", so erfolgt keine Kundentypänderung bei einem Abbruch der jeweiligen Form.
  • <KundenWeiterleitungen>: Dieses Element muss das Attribut Wahrscheinlichkeit besitzen, welches angibt, mit welcher Wahrscheinlichkeit Kunden dieses Typs nach ihrem Gespräch weitergeleitet werden. Ist die Weiterleitungswahrscheinlichkeit größer als 0, so muss das Element ein oder mehrere untergeordnete Elemente vom Typ WeiterleitungenNeuerKundentyp besitzen, die angeben, wie sich der Kundentyp bei einer Weiterleitung ändert. Außerdem kann das KundenWeiterleitungen-Element eines oder mehrere WeiterleitungenSkillLevelSpezifisch-Unterelemnte enthalten, die das allgemeine Weiterleitungsverhalten in Abhängigkeit von dem Skill-Level des Agenten, der den Kunden bedient hat, überschreiben.
    • <WeiterleitungenNeuerKundentyp>: Dieses Element muss einen Parameter Name besitzen, der angibt, welchem neuen Kundentyp Kunden dieses Typs bei der Weiterleitung erhalten können. Als Inhalt des Elements muss die Rate, mit der Weiterleitungen zu dem Typ erfolgen, angegeben werden. (Die Rate muss eine nichtnegative Zahl sein, die auch größer als 1 sein kann. Die Wahrscheinlichkeit für eine Weiterleitung zu dem angegebenen Kundentyp ergibt sich, in dem die jeweilige Rate durch die Summe aller Raten geteilt wird.)
    • <WeiterleitungenSkillLevelSpezifisch>: Dieses Element verhält sich größtenteils identisch wie das Element KundenWeiterleitungen. Es besitzt ein Wahrscheinlichkeit-Attribut und beliebig viele WeiterleitungenNeuerKundentyp-Unterelemente. Zusätzlich muss aber über ein SkillLevel-Attribut angegeben werden, für welches Skill-Level dieser Regelsatz verwendet werden soll. Ist das Gespräch eines Kunden zu Ende, so wird zunächst geprüft, ob es eine passende WeiterleitungenSkillLevelSpezifisch-Regel für den Skill-Level des Agenten, der den Kunden bedient hat, gibt. Wenn ja, wird gemäß dieser Regel bestimmt, ob der Kunde weitergeleitet wird und wenn ja, welchen Kundentyp er erhält. Wenn nein, wird gemäß der unter KundenWeiterleitungen definierten Regeln besimmt, ob der Kunde weitergeleitet wird und wenn ja, welchen Kundentyp er erhält.
  • <KundenWiederanrufer>: Dieses Element muss das Attribut Wahrscheinlichkeit besitzen, welches angibt, mit welcher Wahrscheinlichkeit Kunden dieses Typs nach einem erfolgreichen Anruf einen weiteren Anruf tätigen. Ist die Wiederanrufwahrscheinlichkeit größer als 0, so muss das Element ein Element des Typs WiederanrufAbstaendeVerteilung, welches die Verteilung der Wiederanruf-Abstände angibt und ein oder mehrere untergeordnete Elemente vom Typ WiederanrufNeuerKundentyp besitzen, die angeben, wie sich der Kundentyp bei einem Wiederanruf ändert. Außerdem kann das KundenWiederanrufer-Element eines oder mehrere WiederanrufSkillLevelSpezifisch-Unterelemnte enthalten, die das allgemeine Wiederanrufverhalten in Abhängigkeit von dem Skill-Level des Agenten, der den Kunden bedient hat, überschreiben.
    • <WiederanrufAbstaendeVerteilung>: Gibt die Verteilung der Abstände zwischen dem Auflegen eines Kunden nach einem Gespräch und einem weiteren Anruf desselben Kunden an.
    • <WiederanrufNeuerKundentyp>: Dieses Element muss einen Parameter Name besitzen, der angibt, welchem neuen Kundentyp Kunden dieses Typs bei dem Wiederanruf erhalten können. Als Inhalt des Elements muss die Rate, mit der Wiederanrufe in Form dieses Typs erfolgen, angegeben werden. (Die Rate muss eine nichtnegative Zahl sein, die auch größer als 1 sein kann. Die Wahrscheinlichkeit für eine Kundentyp ergibt sich, in dem die jeweilige Rate durch die Summe aller Raten geteilt wird.)
    • <WiederanrufSkillLevelSpezifisch>: Dieses Element verhält sich größtenteils identisch wie das Element KundenWiederanrufer. Es besitzt ein Wahrscheinlichkeit-Attribut und beliebig viele WiederanrufNeuerKundentyp-Unterelemente. Zusätzlich muss aber über ein SkillLevel-Attribut angegeben werden, für welches Skill-Level dieser Regelsatz verwendet werden soll. Ist das Gespräch eines Kunden zu Ende, so wird zunächst geprüft, ob es eine passende WiederanrufSkillLevelSpezifisch-Regel für den Skill-Level des Agenten, der den Kunden bedient hat, gibt. Wenn ja, wird gemäß dieser Regel bestimmt, ob der Kunde einen Wiederanruf tätigt und wenn ja, welchen Kundentyp er erhält. Wenn nein, wird gemäß der unter KundenWiederanrufer definierten Regeln besimmt, ob der Kunde einen Wiederanruf tätigt und wenn ja, welchen Kundentyp er erhält.
  • <GewinnProKunde>: Als Inhalt dieses Elements muss angegeben werden, wie viel ein erfolgreich bedienter Kunde dieses Typs wert ist.
  • <KostenProAnrufer>: Dieses Element muss die folgenden beiden Attribute besitzen:
    • Abbruch: Gibt die (virtuellen) Kosten, die durch einen Warteabbruch entstehen, an.
    • Wartesekunde: Gibt die (virtuellen) Kosten, die pro Wartesekunde entstehen, an.

Callcenter und Agenten

Das <CallcenterModell> Element kann beliebig viele Elemente (mindestens jedoch eins) vom Typ <ModellCallcenter> enthalten.

Attribute des ModellCallcenter-Elements

  • Name: Gibt den Namen des Callcenter an. Das Name-Attribut muss existieren.
  • Aktiv: Gibt an, ob das im Folgenden definierte Callcneter in der Simulation verwendet werden soll. Aktiv kann die Werte 0 und 1 annehmen. Das Attribut ist optional; wenn es fehlt, wird "1" angenommen.

Unterelemente des ModellCallcenter-Elements

Allgemeine Angaben

Jedes ModellCallcenter-Element muss von jedem der folgenden Elemente jeweils genau eins enthalten:

  • <CallcenterTechnischeBereitzeit>: Als Inhalt dieses Elements muss die technische Bereitzeit, die in dem betrachteten Callcenter bei der Vermittlung eines Kunden an einen Agenten entsteht, angegeben werden. Optional kann das Element noch ein IstWartezeit-Attribut besitzen, welches die Werte 0 oder 1 annehmen kann. Fehlt es, so wird der Wert "1" angenommen. In diesem Fall wird die technische Bereitzeit vom Kunden wie Wartezeit empfunden, was bedeutet, dass er auch in diesem Zeitraum das Warten noch abbrechen kann.
  • <CallcenterScore>: Als Inhalt dieses Elements muss die Basis-Score des Callcenters, die bei der Berechnung der Agenten-Score verwendet werden soll, angegeben werden.
  • <CallcenterAgentenScore>: Dieses Element muss die beiden Unterelemente AgentenScoreFaktorSeitLetztemAnruf und AgentenScoreFaktorLeerlaufAnteil enthalten. Beide Unterelemente müssen jeweils einen Zahlenwert als Inhalt haben, der angibt, wie stark die Bereitzeit in Sekunen seit dem letzten Anruf in die Berechnung der Agenten-Score eingehen soll, bzw. der angibt, wie stark der Bereitzeitanteil des Agenten (in Prozentpunkten) in die Berechnung der Agenten-Score eingehen soll.

Optional kann jedes ModellCallcenter-Element von jedem der folgenden Elemente jeweils genau eins enthalten:

  • <ModellProduktivitaet>: In diesem optionalen Feld werden 48 durch ";" getrennte Werte erwartet, die die Produktivität der Agenten im Tagesverlauf angeben. Die verfügbare Arbeitsleitung pro Halbstundenintervall bei Agentengruppen, die über eine Arbeitsleitungsverteilungskurve oder über die Nachmodellierung der Kundenankünfte definiert sind, wird mit diesem Produktivitätswert multipliziert. Fehlt diese Angabe, so wird der globale Wert verwendet. Die Callcenter-abhängige Produktivität kann optional pro Agentengruppe überschrieben werden.
  • <ModellPlanungsaufschlag>: In diesem Feld werden 48 durch ";" getrennte Werte erwartet, die die angeben, um wie viel höher die planerischer Agentenanzahl sein soll als im Modell im Tagesverlauf angeben. Die verfügbare Arbeitsleitung pro Halbstundenintervall bei Agentengruppen, die über eine Arbeitsleitungsverteilungskurve oder über die Nachmodellierung der Kundenankünfte definiert sind, wird mit diesem Produktivitätswert multipliziert und in der Statistik gespeichert. Die Agentenanzahl inkl. diesem Aufschlag wird nicht für die Simulation selber verwendet. Fehlt diese Angabe, so wird der globale Wert verwendet. Die Callcenter-abhängige Produktivität kann optional pro Agentengruppe überschrieben werden.

Mindestwartezeiten

Jedes <ModellCallcenter>-Element kann optional genau ein <CallcenterMindestWartezeit>-Unterelement enthalten.

Jedes <CallcenterMindestWartezeit>-Element kann beliebig viele <MindestWartezeitKundentyp>-Unterelemente besitzen. Jedes <MindestWartezeitKundentyp>-Element definiert dabei eine Mindestwartezeit für einen Kundentyp in dem jeweiligen Callcenter. (Eine Mindestwartezeit ist die minimale Zeit, die ein Kunde bereits gewartet haben muss, bevor er in das betreffende Callcenter vermittelt werden darf.) Der Kundentyp, für den die Mindestwartezeit gelten soll, muss dabei in dem Attribut Name festgelegt werde. Die Wartezeit selbst wird als Sekundenwert als Inhalt des <MindestWartezeitKundentyp>-Elements angegeben.

Agentengruppen

Jedes <ModellCallcenter>-Element muss ein oder mehrere <CallcenterAgentengruppe>-Unterelemente enthalten. Jedes <CallcenterAgentengruppe>-Element beschreibt eine Agentengruppe mit jeweils bestimmten gemeinsamen Eigenchaften innerhalb des Callcenters.

Jedes <CallcenterAgentengruppe>-Element kann ein Aktiv-Attribut besitzen, welches einen der Werte 0 oder 1 annehmen darf. Fehlt das Aktiv-Attribut, so wird für es der Wert 1 angenommen. Über das Aktiv-Attribut kann bestimmt werden, ob die Agentengruppe bei der Zusammenstellung der Agenten für die Simulation berücksichtigt werden soll (=1) oder nicht (=0).

Agentengruppen können entweder über eine feste Anzahl an Agenten und einen Arbeitzeitbereich oder über eine Verteilung von Agenten-Arbeits-Halbstunden über den Tag oder so, dass ihre Arbeitszeiten bestimmte Kundenankunftsverteilungen nach modellieren, definiert werden:

  • Agenten mit vorgegebenen Arbeitszeiten:
    In diesem Fall muss das <CallcenterAgentengruppe>-Element jeweils ein <AgentengruppeAnzahl>-Unterelement besitzen, welches die Anzahl der Agenten in der Gruppe angibt, und ein <AgentengruppeArbeitszeitbeginn>-Unterelement, welches die Dienstbeginn-Zeit der Agenten dieser Gruppe angibt. Optional kann außerdem noch ein <AgentengruppeArbeitszeitende>-Unterelement angegeben werden, welches die Dienstende-Zeit der Agenten der Gruppe angibt. Fehlt dieses Unterelement, so wird angenommen, dass die Agenten bis zum Ende des Simulationstages arbeiten.
  • Agentengruppe aus vorgegebener Arbeiszeitverteilung:
    In diesem Fall muss das <CallcenterAgentengruppe>-Element ein <AgentengruppeVerteilung>-Unterelement besitzen, welches 24, 48 oder 96 durch Semikolons getrennte Werte ebnthält. Jeder Wert steht dabei für die Agentenanzahl in einem Stunden-, Halbstunden- oder 15-Minuten-Intervall. Der Simulator erzeugt bei Simulationsbeginn aus diesen Werten konkrete Agenten. Bei der Schichtplanung wird als gewünschte Schichtlänge die Anzahl an Halbstundenintervallen verwendet, die in dem Elemente <ModellBevorzugteSchichtlaenge> angegeben ist. Außerdem kann über das Element <ModellMinimaleSchichtlaenge> eine minmale Schichtlänge definiert werden. Existiert eines oder beide Elemente nicht, so wird der jeweilige Wert aus dem gleichnamigen Element im <CallcenterModell>-Element verwendet. Zusätzlich kann das <AgentengruppeVerteilung>-Element noch das Attribut SchichtBisMitternachtIstOpenEnd besitzen. Existiert dieses Attribut und ist es auf 1 gesetzt, so endet die tatsächliche Arbeitszeit der Agenten, die laut Schichtplan bis 24 Uhr arbeiten sollen, nicht um 24 Uhr sondern erst, wenn die letzten Kunden das System verlassen haben.
  • Verteilung über den Tag gemäß Kundenankünften:
    In diesem Fall muss das <CallcenterAgentengruppe>-Element ein <AgentengruppeNachKundenankuenften>-Unterelement besitzen, welches über das Attribut Agentenhalbstunden angibt, wie viele Arbeits-Halbstundenintervalle zur Verfügung stehen und außerdem ein odere mehrere <KundenankuenfteGruppe>-Unterelemente besitzt. Die Verteilung der verfügbaren Agentenarbeitszeit erfolgt gemäß der Ankunftsverteilung eines oder mehrerer Kundentypen. Welche Kundentypen in welchem Verhältnis in die Berechnung einfließen sollen, kann über die <KundenankuenfteGruppe>-Elemente gesteuert werden. Jedes dieser Elemente steht für einen Kundentyp. Der Name des Kundentyps muss als Elementinhalt angegeben werden und über das Attribut Rate muss angegeben werden, wie stark der jeweilige Kundentyp bei der Berechnung berücksichtigt werden soll. Der Simulator erzeugt bei Simulationsbeginn aus diesen Werten konkrete Agenten. Bei der Schichtplanung wird als gewünschte Schichtlänge die Anzahl an Halbstundenintervallen verwendet, die in dem Elemente <ModellBevorzugteSchichtlaenge> angegeben ist. Existiert dieses Element nicht, so wird der Wert aus dem gleichnamigen Element im <CallcenterModell>-Element verwendet.

Unabhängig davon, wie die Arbeitszeiten der Agenten der Gruppe definiert sind, muss jedes <CallcenterAgentengruppe>-Element jeweils genau eines der beiden folgenden Unterelemente besitzen:

  • <AgentengruppeSkillLevel>: Als Textinhalt dieses Elements muss angegeben werden, zu welchem Skill-Level die Agentengruppe gehört.
  • <KostenProStunde>: Als Textinhalt dieses Elements muss angegeben werden, wie teuer eine Arbeitsstunde eines Agents dieses Typs ist.

Optional kann jede Agentengruppe jeweils ein Element der folgenden Art besitzen:

  • <ModellProduktivitaet>: In diesem optionalen Feld werden 48 durch ";" getrennte Werte erwartet, die die Produktivität der Agenten im Tagesverlauf angeben. Die verfügbare Arbeitsleitung pro Halbstundenintervall bei Agentengruppen, die über eine Arbeitsleitungsverteilungskurve oder über die Nachmodellierung der Kundenankünfte definiert sind, wird mit diesem Produktivitätswert multipliziert. Fehlt diese Angabe, so wird der globale Wert verwendet.
  • <ModellPlanungsaufschlag>: In diesem Feld werden 48 durch ";" getrennte Werte erwartet, die die angeben, um wie viel höher die planerischer Agentenanzahl sein soll als im Modell im Tagesverlauf angeben. Die verfügbare Arbeitsleitung pro Halbstundenintervall bei Agentengruppen, die über eine Arbeitsleitungsverteilungskurve oder über die Nachmodellierung der Kundenankünfte definiert sind, wird mit diesem Produktivitätswert multipliziert und in der Statistik gespeichert. Die Agentenanzahl inkl. diesem Aufschlag wird nicht für die Simulation selber verwendet. Fehlt diese Angabe, so wird der globale Wert verwendet.

Desweiteren kann jedes <CallcenterAgentengruppe>-Element beliebig viele Unterelemente des Typs <KostenProKundengruppe> besitzen. Jedes <KostenProKundengruppe>-Element muss die folgenden drei Attribute besitzen:

  • Kundentyp: Gibt an, auf welchen Kundentyp sich der Kostendatensatz beziehen soll.
  • ProAnruf: Kosten pro Anruf des jeweiligen Kundentyps
  • ProMinute: Kosten pro Gesprächs- und Nachbearbeitungszeitminute beim jeweiligen Kundentyp

Skill-Level

Das <CallcenterModell> Element kann beliebig viele Elemente (mindestens jedoch eins) vom Typ <ModellSkillLevel> enthalten.

Attribute des ModellSkillLevel-Elements

  • Name: Gibt den Namen des Skill-Levels an. Das Name-Attribut muss existieren.

Unterelemente des ModellSkillLevel-Elements

Jedes <ModellSkillLevel>-Element muss für jeden Kundentyp, den Agenten des betrachteten Skill-Level bedienen können sollen, ein <SkillLevelKundentyp>-Unterelement enthalten.

Jedes <SkillLevelKundentyp>-Element muss ein Name-Attribut besitzen, in dem angegeben ist, auf welchen Kundentyp es sich beziehen soll. Außerdem muss jedes <SkillLevelKundentyp>-Element von jedem der drei folgenden Unterelemente-Typen jeweils genau eins enthalten:

  • <SkillLevelKundentypBedienzeitverteilung>: Gibt die Bedienzeitverteilung, die bei der Bedienung von Kunden des im Name-Attribute des <SkillLevelKundentyp>-Elements angegebenen Typs durch Agenten mit dem betrachteten Skill-Level verwendet werden soll, an. Zusätzlich zu dem obligatorischen globalen <SkillLevelKundentypBedienzeitverteilung>-Element können weitere <SkillLevelKundentypBedienzeitverteilung>-Elemente existieren, die ein Intervall-Attribut besitzen. Diese Elemente beziehen sich dann auf eines der 48 Halbstundenintervalle (von 1 bis 48 durchnummeriert) und überschreiben den globalen Wert.
  • <SkillLevelKundentypBedienzeitAddOn>: Enthält einen mathematischen Ausdruck, der angibt, wie sich die Wartezeit eines Kunden des im Name-Attribute des <SkillLevelKundentyp>-Elements angegebenen Typs durch Agenten mit dem betrachteten Skill-Level verlängernd auf die Bedienzeit auswirkt. Zusätzlich zu dem obligatorischen globalen <SkillLevelKundentypBedienzeitAddOn>-Element können weitere <SkillLevelKundentypBedienzeitAddOn>-Elemente existieren, die ein Intervall-Attribut besitzen. Diese Elemente beziehen sich dann auf eines der 48 Halbstundenintervalle (von 1 bis 48 durchnummeriert) und überschreiben den globalen Wert.
  • <SkillLevelKundentypNachbearbeitungszeitverteilung>: Gibt die Nachbearbeitungszeitverteilung, die bei der Bedienung von Kunden des im Name-Attribute des <SkillLevelKundentyp>-Elements angegebenen Typs durch Agenten mit dem betrachteten Skill-Level verwendet werden soll, an. Zusätzlich zu dem obligatorischen globalen <SkillLevelKundentypNachbearbeitungszeitverteilung>-Element können weitere <SkillLevelKundentypNachbearbeitungszeitverteilung>-Elemente existieren, die ein Intervall-Attribut besitzen. Diese Elemente beziehen sich dann auf eines der 48 Halbstundenintervalle (von 1 bis 48 durchnummeriert) und überschreiben den globalen Wert.
  • <SkillLevelKundentypScore>: Gibt den Agenten-Score an, den Agenten des betrachteten Skill-Levels für die Bedienung von Kunden des im Name-Attribute des <SkillLevelKundentyp>-Elements angegebenen Typs haben sollen, an.

Warnungsschwellenwerte

Das <CallcenterModell>-Element kann optional ein Element vom Typ <ModellSchellenwerte> enthalten. Das <ModellSchellenwerte>-Element wiederum kann beliebig viele Unterelemente vom Typ <SchellenwertDaten> besitzen.

Attribute des SchellenwertDaten-Elements

Die folgenden Attribute müssen jeweils existieren:

  • Typ: Gibt den Typ des Schwellenwerts an. Dieses Attribut kann einen der folgenden Werte annehmen:
    • Wartezeit auf Anrufbasis
    • Wartezeit auf Kundenbasis
    • Verweilzeit auf Anrufbasis
    • Verweilzeit auf Kundenbasis
    • Erreichbarkeit auf Anrufbasis
    • Erreichbarkeit auf Kundenbasis
    • Service-Level auf Anrufbasis, bezogen auf alle Anrufe
    • Service-Level auf Anrufbasis, bezogen auf erfolgreiche Anrufe
    • Service-Level auf Kundenbasis, bezogen auf alle Kunden
    • Service-Level auf Kundenbasis, bezogen auf erfolgreiche Kunden
    • Auslastung
  • Intervalle: Gibt an, welche Intervalle in welcher Form für den Warnschwellenwert berücksichtigt werden sollen. Es kann "Durchschnitt" angegeben werden, um festzulegen, dass der Durchschnittswert über den gesamten Tag betrachtet werden soll, "Jedes", um festzulegen, dass der Schwellenwert in jedem einzelnen Intervall eingehalten werden soll, oder aber eine Liste von 48 Werten "0" und "1", die angeben, welche Halbstunden-Intervalle betrachtet werden sollen.
  • Gruppen: Gibt an, welche Kundengruppen oder Callcenter in welcher Form für den Warnschwellenwert berücksichtigt werden sollen. Es kann "Durchschnitt" angegeben werden, um festzulegen, dass der Durchschnittswert über alle Gruppen betrachtet werden soll, "Jede", um festzulegen, dass der Schwellenwert für jede Gruppe eingehalten werden soll, oder aber der Name einer Kundengruppe oder eines Callcenters.
  • WarnungGelb: Gibt den Wert an bei dessen Unterschreitung (bei Typen Erreichbarkeit, Service-Level und Auslastung) bzw. bei dessen Überschreitung (Typen Wartezeit und Verweilzeit) eine gelbe Warnung ausgegeben wird. Die Schwellenwerte für Warte- und Verweilzeiten werden dabei als Zeitangaben festgelegt. Die Schwellenwerte für alle anderen Typen in Form von Wahrscheinlichkeiten.
  • WarnungRot: Gibt den Wert an bei dessen Unterschreitung (bei Typen Erreichbarkeit, Service-Level und Auslastung) bzw. bei dessen Überschreitung (Typen Wartezeit und Verweilzeit) eine rote Warnung ausgegeben wird. Die Schwellenwerte für Warte- und Verweilzeiten werden dabei als Zeitangaben festgelegt. Die Schwellenwerte für alle anderen Typen in Form von Wahrscheinlichkeiten.
