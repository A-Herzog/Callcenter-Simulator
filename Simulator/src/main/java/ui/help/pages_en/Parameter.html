Command line parameters

Please note that only the commercial version of the Callcenter Simulators can be controlled by command line parameters.

The general call scheme for the Callcenter Simulator is:

  1. Calling the simulator with a model file as parameter
    <Simulator program file> <model file>
    Starts the graphical user interface of the simulator and loads the given model file.
  2. Calling the simulator with a statistic file as parameter
    <Simulator program file> <statistics file>
    Starts the graphical user interface of the simulator and loads the given statistic file.
  3. Calling the simulator with a command as parameter
    <Simulator program file> <command> <parameter>
    Startet den Simulator im Kommandozeilen-Modus und führt den angegebenen Befehl aus.

Additionally between the name of the program file <Simulator program file> and the parameters can be specified where the simulator configuration is located:

<Simulator program file> cfg="<PathAndFileName>" <OptionalAdditionalParameters>

On program start the configuration file from the progam directory will be loaded first. After this the specified user-defined configuration file will be loaded in addition. When saving the configuration only the user-defined file will be changed.

Commands for the command line mode

Attention: The command line mode is only available if the program is started as a jar file. The Windows exe file does not support the following commands.

The place holder <command> in the third calling variant can be one commands described in the sections below.

At least five additional parameters:
1. Input model file
2. Input table file
3. Output table file
The input files have to be existent, the output has to be non existent. After the first three parameters there can be multiple parameter couples of the form:

  • First table cell (in the Form A1) where the data which are to be loaded are stored
  • Client type name for which the arrivals are to be loaded

  • Agents group number for which the working times are to be loaded

MODELFROMTABLE

Generates a call center model from a clients and an agents table.
The command expects three parameters:
1. Clients input table
2. Agents input table
3. Ouput file name for the model file
The table file names can be extended by a table name inside the workbook file and the name of the first row to be taken into accout both separated by "!".
The caller table can consist of any number of columns. Each column defines one client type. Each column has to consist of the following rows:

  • Name of the client type
  • 48 rows defining the number of calls by half-hour interval
  • Basis priority for the client type
  • Priority change per waiting second
  • Priority change on forwarding
  • Mean waiting time tolerance
  • Retry probability
  • Mean retry time
  • Any number of additional rows defining the probability of clients of this type to be forwarded after the service (in the order of the column order of the table; empty rows mean there is no forwarding to this client type)

The agents table can consist of any number of columns. Each column defines one agents group. Each column has to consist of the following rows:

  • Name of the agents group / the skill level
  • 48 rows for the numbers of agents per half-hour interval
  • Any number of additional rows defining the mean service times for the client types (in the order of the column order of the client types in the callers table; empty rows mean the agents group will not be able to servce clients of this type)

XMLPREPROCESSING

Inserts cells from a table into a XML file.
The command expects 3 parameters:
1. Input model file
2. Table file from where the data is to be loaded
3. Output model file
The input model file can contain expressions like $(TableName!A1) or $(TableName!A1:A48) at any position which indicate that a single cell (A1 in the example) or an area (A1 to A48 in the example) is to be inclueded at this position. If the table name is missing, like $(A1) or $(A1:A48) the first table of the workbook is used.

FILTER

Filter statistic file.
The command expects exactly three additional parameters:
1. Name of the input statistic file
2. Name of the input filter configuration file
3. Name of the output file which will contain the filtered results
If the output file already exists, the new results will be added. Otherwise the file will be created.

IMPORT

Imports data based on a import template from a table.
This command expects exactly four additional parameters:
1. Input model file, from which the model to be changed is loaded
2. Import template file, which specifies which data are to be loaded
3. Imput table file containing the data
4. Output model file

REPORT

Exports the whole report or a part of it.
This command expects exactly three additional parameters:
1. "Inline", Files", "List", "Text", "PDF" or the name of an list entry for
  a) creating a HTML report with embedded images,
  b) creating a HTML report with individual image files,
  c) creating a DOCX report,
  d) creating a PDF report,
  e) getting a list of all available individual documents or
  f) getting an individual document.
2. File name of the input file
3. File name of the output file

MODELINFORMATION

Exports the whole model information or a part of it.
This command expects exactly three additional parameters:
1. "Inline", Files", "List", "Text", "PDF" or the name of an list entry for
  a) creating a HTML report with embedded images,
  b) creating a HTML report with individual image files,
  c) creating a DOCX report,
  d) creating a PDF report,
  e) getting a list of all available individual documents or
  f) getting an individual document.
2. File name of the input file

EXPORT

Export statistic data in a non XML format.
The command expects exactly three additional parameters:
1. Desired output format ("json" or "WebApp")
2. Name of the input statistic file
3. Name of the output file
The input file has to exist, the output file has to be non existent.

Parameters of the individual commands

The additional parameters specified in <parameter> depend on the <command>:

SIMULATION

Start a simulation run.
This command expects exactly two additional parameters:
1. Input model file
2. Output statistic file
The input file has to be existent, the output has to be non existent.

BATCH

Simulates all models in a directory.
The command expects exactly one additional parameter: The name of the directory which contains the model files to be simulated.

CONNECTEDSIMULATION

Simulation with statistic data from the previous simulation day.
Either one or at least three additional parameters. In one parameter is specified it has to be the name of the connected simulation xml file to be executed. If three or more parameters are given they will have the following meaning:
1. Input model file
2. Input statistic file
3. Output statistic file
The input files have to be existent, the output file is not allowed to be existent before the simulation. After these first three parameters there can be any number of pairs of parameters of the following type:

  • Client type for which the cancels calls are to be carried over from the statistic file
  • Part of the cancel calls (as a percent number) that should be carried over as fresh calls

OPTIMIZER

Optimizes a call center model.
This command expects exactly three additional parameters:
1. Input model file
2. Optimizer configuration file
3. Optimizer statistic output file
The input files have to be existent, the output has to be non existent.

OPTIMIZEREXPORT

Extracts the last model of the optimization process.
This command expects exactly two additional parameters:
1. Optimizer statistic file
2. Statiatic output file
The input file has to be existent, the output has to be non existent.

REARRANGE

Rearrange clients or agents to better use the available work force.
This command expects four or more additional parameters:
1. Input model file
2. Output statistic file
3. Rearrange mode
4. Rearrange strength
The input file has to be existent, the output has to be non existent. The rearrange mode has to be "clients" or "agents" and the rearrange strength a number between 0 and 1.
If there are no more additional parameters, all client or agents groups will be rearranged. If there are more parameters, only the groups specified in the 5th, 6th, ... parameter are changed.

PREPLANNING

Erlang C based preplanning for the number of agents.
This command expects exactly two additional parameters:
1. Input model file
2. Results model file
3. Name of the parameter for which a value is to be reached
4. Value of the parameter to be reached
The name of the parameter can be one of the following strings: "Accessibility", "WaitingTime" or "ServiceLevel". Each string can be extended (without a space character) by one or both of the following strings: "Retry" and "Simplified" (in this order). If one of the "...Retry" values is selected, retrying clients will be respected in the Erlang C model. If one of the "...Simplified" values is selected, the model will be simplified to a single skill model before the preplanning is done.

SIMPLIFY

Simplifies the model to a single skill mode.
This command expects two parameters:
1. Input model file
2. Output model file

HEURISTICREVENUEIMPROVEMENT

Changes the numbers of agents to improve the revenue.
This command expects two parameters:
1. Input model file
2. Output model file

Parameters of the individual commands

The additional parameters specified in <parameter> depend on the <command>:

SERVER

Starts the program as simulation server.
This command expects 0 to 3 additional parameters. If these parameters exist they have the following meaning:
1. Port to be used
2. Maximum number of threads to be used per request (0=unlimited, default: 0)
3. Password for encrypted data transfer
If no parameters are specified, the simulator will ask for the port via the console.

LOGFILESERVER

Starts the program as simulation server (writes log results to a file).
This command expects 1 to 4 additional parameters. If these parameters exist they have the following meaning:
1. File name of the log file
2. Port to be used
3. Maximum number of threads to be used per request (0=unlimited, default: 0)
4. Password for encrypted data transfer
If only one parameter is specified, the simulator will ask for the port via the console.

SAASSERVER

Starts the program as SaaS HTML and simulator server.
The command expects exactly one additional parameter: Either the path to the SaaS configuration file or one of the words "Test" or "Demo" to activate the demonstration mode using a default configuration or "Init" to prepare the system for first use.

General commands for setup

CommandDescription
HELPShows this help.
LANGUAGESets up the program language.
MAXTHREADSSets the maximum number of threads used for simulation.
UPDATEChecks for updates and download a new jar file if available.

(If <command> is specified in upper or lower case is not considered.)

Parameters of the individual commands

The additional parameters specified in <parameter> depend on the <command>:

HELP

Shows this help.
This command expects zero or one parameter. If a command line command is specified as additional parameter the help for this command is shown. Otherwise the help for all commands is shown.

LANGUAGE

Sets up the program language.
This command expects as parameter the desired program language: "English" or "German".

MAXTHREADS

Sets the maximum number of threads used for simulation.
The command expects a non negative integer number as parameter.
If "0" is given, the number of simulation threads will not be limited.
If a positive integer number is given, this number is used as the maximum of simulation threads.

UPDATE

Checks for updates and download a new jar file if available.
The command expect as additional parameter the file name for the downloaded update jar file.
